# Claude Settings Pack v0.1

이 저장소는 Claude Code 운영에 필요한 .claude 설정을 프로젝트에 빠르게 이식하고, 팀 단위 AI 개발 플로우를 표준화하기 위한 템플릿입니다.

핵심 목표는 아래 3가지입니다.

- 설정 재사용: 프로젝트마다 반복되는 에이전트/스킬/훅 구성을 공통화
- 작업 표준화: PRD -> Task 분해 -> 구현 -> 테스트 -> 정리 흐름 고정
- 품질 안정화: 역할 분리(coordinator/developer/tester)로 구현과 검증 루프 명확화

---

## 1. 이 저장소로 할 수 있는 것

- .claude 폴더를 새 프로젝트에 그대로 가져와 즉시 운영 시작
- 작업 문서를 Push 단위로 분해해 병렬/순차 실행 전략 수립
- 스킬 기반 개발 규칙(구조, 품질, 성능, SEO 등) 자동 적용
- 훅으로 lint/문서 검증 같은 반복 작업 자동화

---

## 2. 저장소 구성

```text
/
├── README.md
└── .claude/
    ├── README.md
    ├── ARCHITECTURE.md
    ├── behavioral.md
    ├── settings.json
    ├── settings.local.json
    ├── agents/
    ├── roles/
    ├── skills/
    ├── hooks/
    ├── docs/
    └── tasks/
```

각 영역의 역할:

- agents: 단일 세션에서 호출하는 서브에이전트
- roles: 팀 실행 시 역할 분리(coordinator/developer/tester)
- skills: 작업 유형별 규칙과 실행 템플릿
- hooks: 자동 실행 스크립트
- tasks: PRD, todo, done 문서 및 참고 이미지

---

## 3. 빠른 적용 방법

### 방법 A: 복사 방식(가장 단순)

1. 대상 프로젝트 루트에 .claude 폴더를 복사
2. 기존 .claude가 있으면 필요한 파일만 병합
3. 프로젝트 루트에서 Claude Code 실행
4. 샘플 작업으로 task-maker / task-runner 실행

### 방법 B: 서브모듈 방식(여러 프로젝트 공통 관리)

1. 프로젝트에 이 저장소를 submodule로 연결
2. .claude 경로를 기준으로 에이전트가 읽도록 경로 유지
3. 버전 업데이트 시 submodule 버전만 올림

## 4. 권장 AI 개발 플로우

아래는 실무 기준 권장 순서입니다.

### 4-1. 요구사항 정리

- PRD 작성: 범위, 수용 기준, 오픈 이슈를 먼저 고정
- 참고 이미지/문서 경로를 PRD에 명시

### 4-2. 작업 분해

- task-maker로 Push 단위 문서 생성
- 원칙:
  - Push 1개 = 독립 배포 가능한 단위
  - 하위 작업 1개 = 1커밋 단위
  - 모든 구현 작업 뒤에 테스트(T1/T2) 자동 포함

### 4-3. 실행 방식 선택

- 단순 작업: developer 단독
- 중간 이상 작업: coordinator -> developer + tester

### 4-4. 구현과 검증

- developer: 구현 + 최소 변경 + 규칙 준수
- tester: 타입체크/빌드/테스트 + 결함 보고
- coordinator: 작업 재분배/우선순위 조정

### 4-5. 결과 정리

- 완료 문서를 done으로 이동
- result 문서에 변경 요약/검증 결과 기록
- 필요 시 task-cleaner로 일괄 정리

---

## 5. 역할별 운영 기준

### coordinator

- 요구사항 해석, 작업 분배, 진행 관리
- 팀원이 막히는 지점 조기 개입
- 완료 조건 충족 여부 최종 판정

### developer

- 규칙 기반 구현
- 요청 범위 밖 확장 구현 금지
- 커밋 단위로 명확히 결과 제출

### tester

- 타입, 빌드, 테스트 우선 검증
- 재현 가능한 실패 로그 제공
- 사소한 포맷/lint 문제는 즉시 수정

---

## 6. tasks 운영 규칙

권장 패턴:

```text
.claude/tasks/
├── todo/
│   ├── prd-*.md
│   ├── task-YYYY-MM-DD.md
│   └── tasks-*-pushN.md
└── done/
    ├── tasks-*.md
    └── result-*.md
```

운영 팁:

- todo에는 진행 중 문서만 유지
- done에는 완료 문서와 결과 보고만 유지
- 스크린샷/레퍼런스는 tasks 루트에 두고 문서에서 경로로 참조

---

## 7. 품질 가드레일

이 템플릿은 기본적으로 아래 성향을 따릅니다.

- 과도한 추상화보다 단순한 구현 우선
- 요청 범위 밖 변경 금지
- 관련 없는 코드 정리/리팩토링 금지
- 명시적 성공 기준(테스트/검증) 우선

즉, 빠른 구현보다 예측 가능한 변경과 검증 가능한 결과를 우선합니다.

---

## 8. 훅 운영 원칙

- 훅은 빠르고 결정적으로 동작해야 함
- 실패 시 원인을 바로 알 수 있는 메시지 필요
- 로컬 의존성이 큰 훅은 settings.local.json에서 분리

---

## 9. 프로젝트 온보딩 체크리스트

- .claude 구조와 README 설명이 실제와 일치하는가
- roles, agents, skills 이름과 파일명이 일치하는가
- tasks 템플릿 패턴이 팀 운영 방식과 맞는가
- 훅이 현재 프로젝트 런타임에서 정상 동작하는가
- 불필요한 프로젝트 고유 문구가 남아 있지 않은가

---

## 10. 자주 겪는 문제

### 문제 1: 스킬이 자동 적용되지 않음

확인 사항:

- SKILL.md description에 트리거 키워드가 포함되었는지
- 스킬 폴더명과 내부 name이 일치하는지
- 파일 경로가 .claude/skills/<name>/SKILL.md 형태인지

### 문제 2: 작업 문서가 오래 누적됨

해결:

- 주기적으로 done 정리
- task-cleaner를 배치 실행
- task 인덱스 파일에 상태 컬럼 유지

### 문제 3: 역할 분리가 오히려 느림

해결:

- 1~2파일 수정은 developer 단독으로 전환
- 3파일 이상/검증 복잡 작업만 팀 실행

## 11. 버전 운영

권장 방식:

- 태그로 버전 관리 (예: v0.1, v0.2)
- 변경 이력은 CHANGELOG 또는 릴리스 노트로 관리
- 파괴적 변경은 마이너/메이저 업

## 12. 라이선스/보안 주의

- 민감정보(토큰, 키, 내부 URL)는 .claude에 직접 커밋 금지
- settings.local.json은 개인/로컬 값 전용
- 외부 공유 전 tasks 안의 내부 자료(이미지/문서) 노출 범위 확인
